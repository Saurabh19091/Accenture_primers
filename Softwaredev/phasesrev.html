SRS (Software Requirements Specification) — Template & Example

Project: Student Management System (SMS) — brief overview and key sections of an SRS.

1. Introduction

  • Purpose: Automate student records, marks, attendance and reporting for St. Peters University.
  • Scope: Student enrollment, courses, grading, transcripts, admin dashboards, role-based access.

2. Functional Requirements (sample)

IDRequirementPriority
FR-01Admin can create/update/delete student recordsHigh
FR-02Teacher can enter marks for students per subjectHigh
FR-03System computes GPA and grades automaticallyHigh
FR-04Students can view their transcriptMedium

3. Non-Functional Requirements

  • Performance: 500 concurrent users supported, response time < 2s for common queries.
  • Security: Role-based access, password hashing, HTTPS.
  • Availability: 99.5% uptime.

4. Use Cases (brief)

  1. Enroll Student — Admin enters personal and course details → Student record created.
  2. Enter Marks — Teacher selects class & subject → submits marks → system calculates grade.
  3. View Transcript — Student logs in → views/downloads PDF transcript.
Tip: Keep the SRS concise but traceable — each FR should map to test cases and design modules.

Entity-Relationship Diagram (ERD) — Example for SMS

Entities: Student, Course, Enrollment, Teacher, Grade

ERD tip: Mark primary keys (PK) and foreign keys (FK) clearly. Use an association (Enrollment) for many-to-many between Student & Course.

Design — HLD & LLD (High-level & Low-level)

HLD (High Level)

  • Architecture: 3-tier (Client — Web Server — Database)
  • Modules: Authentication, Student Management, Course Management, Marks Processing, Reports
  • Data Stores: Relational DB (Postgres/MySQL), Redis for caching

LLD (Low Level)

// Sample function (pseudo-JS) to compute grade from marks
function computeGrade(marks){
  if(marks >= 90) return 'A+';
  if(marks >= 80) return 'A';
  if(marks >= 70) return 'B';
  if(marks >= 60) return 'C';
  return 'F';
}
LLD contains data structures, APIs, class diagrams and sequence diagrams for key flows (e.g., marking workflow).

Testing — Strategy & Sample Test Cases

Types to include

  • Unit tests (computeGrade, validation)
  • Integration tests (API + DB)
  • System tests (end-to-end flows)
  • UAT (with actual users/teachers)
  • Performance & Security tests

Maintenance & Deployment

  • Deployment: CI/CD pipeline, blue-green or rolling deploys
  • Monitoring: Prometheus/Grafana, logs, alerts
  • Maintenance types: Corrective, Adaptive, Perfective, Preventive
Plan for backups, schema migrations, and clear rollback strategy in case of faulty releases.

Quick Revision Notes

SRS: Purpose, Scope, FR, NFR, Use Cases, Traceability Matrix.
ERD: Identify entities, PK, FK, relationships (1-1,1-M,M-M), normalize to 3NF.
Design: HLD = modules & tech choices; LLD = functions, data structures, API signatures.
Coding: Keep logic testable, follow SOLID, use migrations & linting.
Testing: Unit → Integration → System → UAT. Map tests to SRS IDs.
Deployment: CI/CD, monitoring, automated backups.
